# Чистый REST: Примеры запросов и ответов

Если бы мы переделывали архитектуру под Чистый REST (Вариант 2), то вместо одного "жирного" запроса к `/spa/api/projects/{id}`, фронтенд должен был бы сделать серию атомарных запросов к разным эндпоинтам.

Ниже приведены примеры того, как выглядели бы запросы и ответы в такой архитектуре.

---

## 1. Запрос базовой информации о проекте
Фронтенд запрашивает только сам проект. Никаких досок и участников здесь нет.

**Запрос:**
`GET /spa/api/projects/26`

**Ответ (200 OK):**
```json
{
  "id": 26,
  "name": "Название проекта",
  "description": "Описание",
  "createdAt": "2026-06-30T10:00:00Z",
  "updatedAt": "2026-06-30T10:00:00Z",
  "ownerId": 144
}
```

---

## 2. Запрос списка досок проекта
Фронтенд делает отдельный запрос, чтобы получить список досок, привязанных к проекту 26.

**Запрос:**
`GET /spa/api/projects/26/boards`

**Ответ (200 OK):**
```json
[
  {
    "id": 1,
    "title": "Главная доска",
    "position": 1.0,
    "createdAt": "2026-06-30T10:00:00Z",
    "updatedAt": "2026-06-30T10:05:00Z"
  }
]
```

---

## 3. Запрос участников проекта
Фронтенд делает еще один запрос, чтобы получить список всех пользователей, у которых есть доступ к проекту.

**Запрос:**
`GET /spa/api/projects/26/members`

**Ответ (200 OK):**
```json
[
  {
    "userId": 144,
    "role": "KANBAN_ADMIN",
    "position": 0
  },
  {
    "userId": 1745,
    "role": "KANBAN_EDITOR",
    "position": 0
  }
]
```

---

## 4. Запрос данных пользователей (Опционально)
В чистом REST микросервис канбана может не отдавать персональные данные (Имя, Фамилию, Аватарку), так как он отвечает только за канбан. В идеальном мире фронтенд должен пойти в сервис пользователей (User Service / Identity Provider), чтобы обогатить данные.

**Запрос (в сервис пользователей):**
`GET /api/users?ids=144,1745`

**Ответ (200 OK):**
```json
[
  {
    "id": 144,
    "login": "ivanov_ii",
    "lastname": "Иванов",
    "firstname": "Иван",
    "avatarUrl": "https://..."
  },
  {
    "id": 1745,
    "login": "petrov_pp",
    "lastname": "Петров",
    "firstname": "Петр",
    "avatarUrl": "https://..."
  }
]
```

---

## 5. Запрос структуры конкретной доски
После того, как фронтенд получил список досок (Шаг 2), он загружает колонки и карточки активной доски. В чистом REST это тоже может быть разбито на части:

**Запрос колонок:**
`GET /spa/api/boards/1/columns`

**Ответ (200 OK):**
```json
[
  {
    "id": 1,
    "title": "В работе",
    "headerColor": "bg-primary",
    "position": 1.0
  }
]
```

**Запрос карточек в колонке:**
`GET /spa/api/columns/1/cards`

**Ответ (200 OK):**
```json
[
  {
    "id": 10,
    "title": "Название карточки",
    "priority": "high",
    "position": 1.0
  }
]
```

---

### Вывод
Как видите, в **Чистом REST** структура API невероятно красивая, логичная и атомарная. Каждый эндпоинт отвечает только за одну таблицу/сущность. 

Но для отображения одной страницы Канбан-доски фронтенду придется сделать **от 4 до 7 последовательных или параллельных запросов**. А при подходе **BFF ("Толстое API")**, фронтенд делает всего **один** запрос к бэкенду.
